# 8회차 최종 프로젝트 — 가이드

## 1. 가정 환경 (Persona)

> **도메인은 자유** — 광고 / 이커머스 주문 / SaaS 사용 로그 / IoT 센서 / 게임 로그 / 금융 거래 등 본인이 익숙하거나 관심 있는 도메인, 목표하는 기업의 도메인을 선택하면 됩니다. 스터디 시간에 다룬 내용을 가지고 더 개선해보는 방향으로 진행하셔도 좋습니다.

### 데이터 규모와 패턴
- 현재 **일 100만 이벤트** (시간당 ~4만, 초당 ~12)의 작은 환경 가정.
- 평균 이벤트 크기 ~1KB → **일 ~1GB raw, 월 ~30GB, 연 ~365GB**.
- 트래픽 패턴: **피크 시간대 평소의 2~3배**, 주말 1.5배 (도메인에 맞게 직접 정의).
- **6개월 내 10x 성장 예상** — 일 1,000만 이벤트가 현실적 목표.
- **장기 100x 가능성** — 1~2년 내 일 1억 이벤트(일 ~100GB)까지 갈 수 있다고 가정 (스케일 아웃 설계 사고의 근거).

### 기술 스택 (자유 선택 — 강의 다룬 범위가 기본)
- 클라우드: AWS 가정 (S3, Glue Catalog, EMR / EKS Spark, Athena).
- BI: QuickSight / Superset / Metabase 등 자유.
- 오케스트레이션: Airflow / cron / GitHub Actions 등 자유.

## 2. 필수 구현 요건 (4가지)

발표와 README 모두에서 다음 4가지가 명확히 보여야 합니다:

1. **Iceberg 테이블 활용**
2. **메달리온 아키텍처** — Bronze (raw) / Silver (정제) / Gold (집계) 3계층 분리.
3. **Iceberg 매니지먼트 자동화** — Compaction / Expire Snapshots / Orphan Cleanup 중 **최소 1개 이상** Airflow / cron 등으로 자동화.
4. **대시보드** — BI 도구는 자유. 비즈니스 KPI와 운영 메트릭 모두 포함.

## 3. 평가 기준

### ① 운영 가시성

> *"운영자가 매일 5분 안에 파이프라인 헬스체크를 할 수 있는가?"*

**Iceberg 메타 테이블(`snapshots` / `files` / `history` / `manifests`) 기반 헬스 쿼리 5~10개** + **BI 대시보드 운영 메트릭 탭** (신선도, 일자별 행 수, 파일 수·평균 크기).

### ② 스케일 인/아웃 사고력

> *"10배를 넘어 100배 성장 (일 100만 → 일 1억)이 일어나면, 어디가 깨질것이고  어떻게 스케일 아웃해야할 것인가?"*

실제 구현은 안해도되고 어떻게 할건지에 대해서만 정리해보면 됩니다. 이러한 질문에 답변할 수 있는가?가 중요한 부분이 될 거 같습니다.

### ③ Iceberg 필요성 이해

> *"왜 그냥 Parquet + Glue가 아니라 Iceberg인가?"*


### ④ 협업 · 지속 가능성

> *"6개월 후 새 팀원이 이 레포에 들어와 다음 기능을 자연스럽게 추가할 수 있는가?"*

완벽하게 돌아가는 *일회성* 파이프라인도 좋지만, **계속 가꿔나갈 수 있는 구조**가 더 중요합니다. 누군가와 협업하는 프로젝트를 만든다는 자세로 프로젝트 진행해주시면 좋을거 같습니다.

## 4. 장애·운영 시나리오 (예시)

다루면 운영 사고력과 Iceberg 활용이 잘 드러나는 시나리오들입니다. **본인 도메인에 맞는 다른 시나리오로 대체하거나, 일부만 다뤄도 무방.**  *"어떻게 대응하는가"* 를 설명해 주시거나 구현해주시면 됩니다. 

1. **Spark Streaming Job이 새벽에 OOM으로 죽었다** — 재시작 시 Kafka offset 복구(`checkpointLocation`), 중복 방지(멱등성·`event_id` 유니크 키), raw zone에서의 재처리 가능성.
2. **3개월치 정제 로직 버그 발견 → 백필 필요** — raw 보존, MERGE 멱등성, 백필 중 대시보드 일관성, Expire/Orphan 정책이 백필 윈도우를 가로막지 않는가.
3. **컴팩션 도중 streaming MERGE가 같은 파티션을 건드린다** — Iceberg OCC가 어떻게 감지·중재하는가, 충돌 회피 운영 패턴(시간대 분리 / `partial-progress.enabled`).

## 5. 권장 산출물 형식

### GitHub 레포 디렉토리 구조

```
final-project/
├── README.md           # 평가 답변 섹션 (아래 헤더 참고)
├── infra/              # 환경 구성 (택 1) — AWS IaC (Terraform/CDK/CloudFormation) | docker-compose | setup 스크립트
├── code/
│   ├── ddl/            # Bronze / Silver / Gold DDL
│   ├── pipelines/      # 정제 · 집계 · MERGE 로직
│   └── health-queries/ # 운영 헬스 쿼리 5~10개 (.sql 또는 notebook)
├── orchestration/      # 스케줄링·워크플로우 (Airflow DAG / cron / GitHub Actions 등) — 의존성·스케줄·재시도/백필 정책
├── dashboard/          # BI 대시보드 스크린샷 (+ 가능하면 정의 파일)
└── slides.pdf          # 10–15분 발표 자료 (있어도 되고 없어도 됨)
```

> **로컬 Docker 환경도 OK** (Spark + MinIO + Hive Metastore + Trino/DuckDB 등). README **전체 아키텍처** 섹션에 *"AWS 환경 vs 로컬 Docker 환경"* 중 무엇을 썼는지 설명
>
> **심화 (선택)**: 운영 메트릭을 BI 대시보드 운영 탭이 아니라 별도로 분리하고 싶다면 **Grafana + Prometheus** 같은 모니터링 스택을 `monitoring/` 디렉토리로 추가해도 좋음.

```markdown

## 6. 대시보드 (스크린샷 + 운영 메트릭)

```
